Overview

What is Zitadel?

  • Open-source identity & access management
  • OIDC, OAuth2, SAML2, passkeys, MFA — out of the box
  • API-first, runs on Kubernetes or bare metal
  • Postgres is the only external dependency
  • SaaS alternative to Auth0 — self-hosted, no per-MAU pricing
  • Modern alternative to Keycloak — API-first, built-in multi-tenancy
Architecture

What we're building

Docker → k3d → Traefik → Zitadel → PostgreSQL
└── Login UI
  • k3d runs k3s inside Docker — no VMs, Traefik built in
  • Traefik configured via a mounted config file — no separate install
  • Zitadel deploys two pods: core API and Login UI
  • PostgreSQL as a Helm dependency

The demo

The Makefile

make -C demo deploy   # spin everything up
make -C demo destroy  # tear it all down

deploy runs two steps under the hood:

init:
	k3d cluster create zitadel-demo \
		--port "80:80@loadbalancer" \
		--port "443:443@loadbalancer" \
		--volume "$(pwd)/traefik-config.yaml:/var/lib/rancher/k3s/server/manifests/traefik-config.yaml@server:0"

install:
	helm upgrade --install zitadel zitadel/zitadel \
		--values zitadel-values.yaml \
		--wait --timeout 10m

Next

Running Zitadel
in Production

The demo is up. Here's what changes when you go to production.

Production

Database

The bundled PostgreSQL chart is for local use only — in production, point Zitadel at any Postgres-compatible database:

  • Postgres Helm chart — self-managed
  • Postgres Operator — CloudNativePG or Zalando
  • AWS RDS / Aurora
  • Google Cloud SQL
  • Azure Database for PostgreSQL
CockroachDB support was dropped in Zitadel v3
Production

Ingress

Zitadel must run behind an ingress controller — TLS terminates there, and the ingress forwards h2c (HTTP/2 cleartext) traffic to the pods. Both the core pod and Login UI need their own ingress rules.

  • Traefik — handled automatically, the chart adds the h2c annotation by default
  • Other ingress controllers must be configured manually — each has its own quirks
# Traefik
ingress:
  enabled: true
  className: traefik
login:
  ingress:
    enabled: true
    className: traefik

¹ Gateway API support added in chart v9.26.0 — github.com/zitadel/zitadel-charts/releases/tag/zitadel-9.26.0

Production

TLS

TLS is terminated at the ingress controller — Zitadel speaks plain HTTP/2 internally. You need a valid certificate on your ingress; how you get it depends on your setup.

  • cert-manager — the universal option, works with any ingress controller. Supports Let's Encrypt, Vault, Venafi, and more
  • Ingress-native — some controllers handle certs themselves: Traefik has built-in ACME, AWS ALB uses ACM, GKE uses Google-managed certs
  • Bring your own — manually create a Kubernetes TLS secret and reference it in the ingress
Production

Secret Management

Never store secrets in values.yaml — use External Secrets Operator (ESO)¹ to pull from your existing secret store. Supports AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, Vault, and more.

  • Master key: inline (zitadel.masterkey) or reference an existing secret (zitadel.masterkeySecretName)
  • Master key cannot be changed after first run — store it safely
  • Use ESO for all secrets: master key, DB credentials, and any other sensitive config
  • Never commit secrets to Git

¹ external-secrets.io/latest/provider

Production

Pod Security

Principle of least privilege — Zitadel pods should only have the permissions they need to run, nothing more. Configured in values.yaml and applies to both the core and Login UI pods.

  • podSecurityContext — pod-level settings, applied before any container starts
  • securityContext — container-level settings, applied to each container
podSecurityContext:
  runAsNonRoot: true          # refuse to run as root
  runAsUser: 1000             # run as unprivileged user
  fsGroup: 1000               # volume mounts owned by this group
  seccompProfile:
    type: RuntimeDefault      # restrict syscalls to the runtime default

securityContext:
  runAsNonRoot: true          # redundant but explicit — belt and braces
  runAsUser: 1000             # container process UID
  readOnlyRootFilesystem: true        # no writes to the container filesystem
  privileged: false                   # no access to host devices
  allowPrivilegeEscalation: false     # cannot gain more privileges than parent process
  capabilities:
    drop:
      - ALL                   # drop all Linux capabilities
Production

Rate Limiting

Zitadel does not rate limit itself — configure it at the ingress layer to protect auth endpoints from brute-force attacks. Traefik supports this natively per source IP, no WAF required.

  • Counts requests per client IP — excess requests are dropped
  • Applied via a RateLimit middleware attached to the Zitadel ingress
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: zitadel-rate-limit
spec:
  rateLimit:
    average: 10   # requests per second per IP
    burst: 20     # allowed burst above average
Production

Network Policies

Use Network Policies to limit blast radius if any pod is compromised. Applied separately from the Helm chart; requires a CNI that supports NetworkPolicy (Calico, Cilium).

  • Core pod: allow ingress from Traefik, egress to PostgreSQL only
  • Login UI pod: allow ingress from Traefik, egress to the core pod only
# Core pod
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: zitadel
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: zitadel
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app.kubernetes.io/name: traefik
  egress:
    - to:
        - podSelector:
            matchLabels:
              app.kubernetes.io/name: postgresql
# Login UI pod
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: zitadel-login
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: zitadel-login
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app.kubernetes.io/name: traefik
  egress:
    - to:
        - podSelector:
            matchLabels:
              app.kubernetes.io/name: zitadel
Production

High Availability

Zitadel is stateless and scales horizontally — configure both the core and Login UI deployments independently or the Login UI becomes a single point of failure.

  • HPA overrides replicaCount when enabled
  • minAvailable and maxUnavailable are mutually exclusive in pdb
# Main Zitadel container
replicaCount: 3
pdb:
  enabled: true
  minAvailable: 1
topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
zitadel:
  autoscaling:
    enabled: true
    minReplicas: 3
    maxReplicas: 10
    targetCPU: 80

# Login UI container
login:
  replicaCount: 3
  pdb:
    enabled: true
    minAvailable: 1
  topologySpreadConstraints:
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule
  autoscaling:
    enabled: true
    minReplicas: 3
    maxReplicas: 10
    targetCPU: 80
Production

Caching

Add a shared Redis or Valkey cache to reduce database load and keep session state consistent across replicas. Any Redis-compatible service works.

  • Self-hosted Redis or Valkey
  • AWS Elasticache
  • Redis Enterprise
  • Upstash
  • Google Cloud Memorystore
ZITADEL_CACHES_CONNECTORS_REDIS_ENABLED=true
ZITADEL_CACHES_CONNECTORS_REDIS_URL=redis://user:password@redis:6379/0
# Use rediss:// for TLS
Production

Observability

Zitadel emits all three OpenTelemetry signals out of the box — no instrumentation required.

  • Traces, metrics, and logs — all three signals via standard OTLP
  • Log export is newly added — no Fluentd, no Vector, no log shipper needed
  • Works with any OpenTelemetry-compatible backend (Grafana, Datadog, Honeycomb, etc.)
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317
OTEL_EXPORTER_OTLP_PROTOCOL=grpc
OTEL_SERVICE_NAME=zitadel
Both the core pod and the Login UI pod generate telemetry — configure these env vars on both deployments.¹
